# Мастер-класс по рефакторингу бэковых тестов при помощи ИИ-агентов

На данном мастер-классе мы будем исследовать возможности ИИ-агентов для рефакторинга бэковых автотестов на примере веб-калькулятора с REST API. Будем делегировать нейросетям рутину: исправление техдолга, генерацию сложных структур и массовый рефакторинг — всё это на примере плагина Roo Code для прямого взаимодействия с кодом в IDE. Мы разберем два ключевых подхода: итеративное выполнение мелких задач и планирование реализации под контролем оркестратора. Все задания будут выложены в этот репозиторий после мастер-класса.

## Подготовка и настройка окружения

### 1. Настройка VSCode и RooCode

Подробные инструкции по установке и настройке VSCode, RooCode и провайдера моделей на примере Cerebras AI см. в [README прошлой версии мастер-класса](https://github.com/fortrest-jr/sqa37-calc-task#подготовка-и-настройка-окружения).

**Важно**: Сейчас Cerebras AI не поддерживается в RooCode, поэтому нужно подобрать альтернативный провайдер моделей. Инструкция из прошлой версии универсальна — процесс настройки аналогичен для любого провайдера. Для повседневного использования рекомендую Sourcecraft.

Так же можно воспользоваться [альтернативными IDE](#альтернативные-ide-и-инструменты) со встроенными ИИ-агентами.

### 2. Получение кода проекта

1. Проверьте, установлен ли Git, набрав в консоли:
    ```bash
    git --version
    ```

    Если нет — [установите git](https://git-scm.com/downloads).

2. Клонируйте репозиторий с проектом:

    ```bash
    git clone https://github.com/fortrest-jr/sqa38-calc-task2.git
    cd sqa38-calc-task2
    ```


### 3. Настройка окружения проекта
Выполняется после клонирования репозитория, в созданной папке.

1. Убедитесь, что у вас установлен uv, набрав в консоли:
    ```bash
    uv --version
    ```
    Если нет, [установите uv](https://docs.astral.sh/uv/getting-started/installation/)

2. Установите зависимости:
    ```bash
    uv python install
    uv venv
    uv pip install -r requirements.txt
    ```

3. Откройте в VSCode папку проекта:
    ```bash
    code .
    ```

## Задания для мастер-класса

Все задания предполагается выполнять в плагине Sourcecraft или Roo Code в VSCode, но можно использовать и другие альтернативы.

Следите за размышлениями модели и останавливайте, если что-то идёт не так, например модель путается или думает о чём-то не том.
Вполне нормально писать уточняющие сообщения в любой момент размышлений; возвращаться к предыдущим сообщениям, чекпоинтам и переделывать части.

Кейсы независимы друг от друга — можно выполнять в любом порядке. Все действия стараемся делать через модель без ручных правок. Примеры промптов — подсказки. Пробуйте придумать их самостоятельно.


### Кейс 1. Генерируем сложные структуры (Pydantic-модели для тестов)

**Контекст:** В коде калькулятора история описывается сложными вложенными Marshmallow-схемами для валидации, но они не подходят для тестов.

**Проблема:** Ручное описание JSON для API-тестов — скучно и чревато ошибками. Посмотрите на тест `test_history_endpoints` — там десятки строк ручных проверок вложенной структуры через `has_entry`/`has_entries`/`has_key`.

**Делаем:** Дата-классы (Pydantic-модели) для валидации ответов API в тестах.

**Подход:** Итеративный — делаем маленькими шагами, контролируя результат на каждом этапе. Не мигрируем всё сразу, а начинаем с одной простой модели, убеждаемся что подход работает, и только потом переходим к следующей группе.

Можно использовать не очень умные модели, важен ваш личный контроль результата, стараемся понять как лучше давать задачи и к чему это приводит.

#### Подготовка

Изучите контекст проекта перед началом работы:
* Найдите файл со схемами ответов и разберитесь, какие группы моделей существуют
* Выделите группы по сложности: простые схемы, например, бинарные операции и сложные (история с глубокой вложенностью). Обратите внимание, что для схемы округления нет тестов в `test_api.py` — она не подходит для первой итерации
* Задайте ИИ-агенту уточняющие вопросы, если он предлагает что-то лишнее или упускает вложенные структуры

<details>
<summary>Пример промпта (режим Ask)</summary>

```
У нас в коде используются схемы для валидации: {файл со схемами}
Мы хотим для тестов сделать pydantic-модели
Расскажи, какие модели нужно создать и как они соотносятся со схемами
```
Подставьте путь к файлу со схемами ответов.
</details>

#### Шаги

1. **Первая итерация: создайте простую модель**
    * Начните с самой простой схемы — ответа бинарной операции (4 поля, без вложенности)
    * Создайте для неё Pydantic-модель в отдельной папке для тестовых утилит
    * Проведите ревью сгенерированного кода — соответствует ли модель схеме?
    * Проверьте достаточно ли импортов? Указано ли в зависимостях?
    * Если чего-то не хватает просите агента доработать нужные моменты

    <details>
    <summary>Пример промпта (режим Code)</summary>

    ```
    У нас в коде используются схемы для валидации: {файл со схемами}
    Мы хотим для тестов сделать pydantic-модели

    Создай в папке {папка для моделей} модели для {копируемая модель}
    ```
    Подставьте путь к файлу со схемами, папку для тестовых моделей и название простой схемы.

    * В итоге pydantic должен быть установлен агентом и добавлен в requirements.txt
    </details>

2. **Внедрите простую модель в тесты**
    * Замените ручные проверки в тестах бинарных операций на парсинг через Pydantic
    * Убедитесь, что не потерялись проверки конкретных значений — Pydantic валидирует структуру, но проверки значений всё ещё нужны
    * Запустите тесты и убедитесь, что всё проходит — это ваш эталон для следующих итераций

    <details>
    <summary>Пример промпта (режим Code)</summary>

    Если используется та же задача, то можно не добавлять контекст, а продолжать работать тут же, добавляя лишь задачу.
    
    ```
    Добавь использование модели в ассёрты в тестах {файл с API-тестами}
    ```
    Подставьте файл с API-тестами.
    </details>

3. **Вторая итерация: создайте сложную модель**
    * Переходите к самой сложной схеме — ответу истории с глубокой вложенностью (мета-данные, пагинация, элементы истории с операциями и результатами)
    * **Важно:** передайте ИИ-агенту уже созданную простую модель как пример — это поможет ему выдержать единый стиль
    * Проверьте, что созданы все вложенные структуры (операнды, результат, метаданные элемента, пагинация, пользователь, временные метки, статус)

    <details>
    <summary>Пример промпта (режим Code)</summary>

    ```
    У нас в коде используются схемы для валидации: {файл со схемами}
    Мы делаем для тестов pydantic-модели
    Посмотри пример pydantic-модели: {файл с готовой моделью}
    Создай по аналогии модели для {сложная схема}
    ```
    Подставьте файлы и название сложной схемы.
    </details>

4. **Внедрите сложную модель в тесты**
    * Замените ручные проверки в тесте истории на парсинг через Pydantic
    * Обратите особое внимание на тест `test_history_endpoints` — там самая большая выгода от перехода на модели
    * Запустите тесты и убедитесь, что всё проходит

    <details>
    <summary>Пример промпта (режим Code)</summary>
    Аналогично шагу 2
    </details>

5. **Финальное ревью**
    * Используйте другую модель (не участвовавшую в кодинге) для ревью — нет ли потерянных проверок по сравнению с оригиналом?
    * Можно спросить оценку несколько раз и попросить выявить потенциальные проблемы

    <details>
    <summary>Пример промпта (режим Ask, другая модель)</summary>

    ```
    Проверь тесты в {файл с API-тестами} — корректно ли используются Pydantic-модели?
    Нет ли потерянных проверок по сравнению с оригиналом?
    ```
    </details>

### Кейс 2. Умная массовая замена (миграция hamcrest → pytest)

**Контекст:** В коде тестов калькулятора используется сторонняя библиотека для ассёртов — hamcrest.

**Проблема:** Сложнее развивать фишки pytest, бойлерплейт, лишняя зависимость.

**Делаем:** Чистые ассёрты фреймворка (pytest).

Используйте умные модели, лучше всего с включенным режимом размышления.

#### Шаги

1. **Составьте план миграции**
    * Определите все файлы с тестами
    * Попросите ИИ-агента составить план миграции ассёртов с hamcrest на чистый python
        * Подскажите сохранить его в файл, если агент не сделал этого сам.
    * **Проведите подробное ревью плана** — разберитесь в каждом пункте, уточните все неясные моменты и потенциальные риски которые подсветил ИИ. Не переходите к реализации, пока не будете уверены в плане
    * Можно просить вносить дополнительные правки, которые посчитаете полезными

    <details>
    <summary>Пример промпта (режим Architect)</summary>

    ```
    У нас в коде тестов используются ассёрты на hamcrest: {файлы с тестами}
    Мы хотим мигрировать их на чистые pytest-ассёрты

    Составь план миграции
    и приведи сравнительную таблицу старых и новых вариантов ассёртов
    ```
    Подставьте файлы с тестами.

    Рекомендации при ревью плана:
    * Попросите сравнительную таблицу старых и новых вариантов ассёртов, если её нет
    * Обратите внимание на нетривиальные замены: `close_to` → `pytest.approx`, `calling(...).raises(...)` → `pytest.raises`, вложенные матчеры в `has_entries`
    * Полезно добавлять проверки успешности в конец каждого из шагов, например запуск изменённых тестов
    </details>

2. **Реализуйте план**
    * Попросите модель реализовать план из файла
    * Наблюдайте что происходит, но не влияйте на процесс. Задача эксперимента — составлять правильные планы чтобы агент мог сам их реализовать. 
        * Если что-то пошло плохо — попробуйте убрать все изменения и скорректировать план в 1 шаге, потом снова переходите к реализации. 
        * Но главное в целом — почувствовать как удобнее работать лично вам. Так что если будет удобно вмешиваться и править для хорошего результата — то можно)

    <details>
    <summary>Пример промпта (режим Orchestrator)</summary>

    ```
    Реализуй план {файл с планом миграции}
    ```
    Подставьте файл с планом.
    </details>

3. **Проверьте результат**
    * Не рекомендую вручную запускать тесты для проверки, это обязательно должно было быть сделано в рамках реализации
    * Проведите полное и подробное ревью результата
    * Используйте другую модель (не участвовавшую в планировании и кодинге) для ревью
    * Можно спросить оценку несколько раз и попросить выявить потенциальные проблемы

    <details>
    <summary>Пример промпта (режим Ask, другая модель)</summary>

    ```
    Проверь миграцию тестов с hamcrest на pytest в {файлы с тестами}
    Все ли ассёрты корректно заменены? Нет ли потерянных проверок?
    ```

    * Будет хорошо если в итоге удален hamcrest из venv и requirements.txt
    </details>

## Дополнительная информация

### Альтернативные IDE и инструменты

* **Sourcecraft** — Альтернатива RooCode от Яндекса с мощной бесплатной квотой: https://sourcecraft.dev/portal/code-assistant/. Результат работы ИИ мне нравится больше, но скорость ниже.

* **Веб-IDE без подготовки окружения**, если нет возможности настроить локальное:
    * **Firebase Studio** — https://studio.firebase.google.com/
        * IDE прямо в браузере, выглядит как VSCode
        * Управляется почти так же, как на мастер-классе
        * Сложнее в настройке, чем Replit (описан ниже)
        * Для подготовки:
            1. Импортировать этот репозиторий, вставив текущую ссылку на GitHub
            2. Вместо установки uv и python нужно разрешить студии добавить файл dev.nix
                * Потом открыть его и внутрь массива `packages` добавить строки:
                    ```nix
                    pkgs.uv
                    pkgs.python313
                    ```
            3. Настроить окружение по инструкциям выше, начиная с команды `uv venv`
    
    * **Replit**:
        * Работает и с мобилки
        * Интерфейс сильно отличается
        * Для подготовки: ремикснуть по ссылке https://replit.com/@arssobolev/sqa38-calc-task2

### Расширение возможностей агентов

Для углублённого изучения и расширения возможностей RooCode и аналогов:
* **[Agents.md](https://docs.roocode.com/features/custom-instructions)** — Документация по созданию и управлению ИИ-агентами. Позволяет универсально определять поведение и стратегии агента.
* **[Skills](https://docs.roocode.com/features/skills)** — Система расширения функциональности агентов. Можно создавать собственные skills для специфических задач и интегрировать их в рабочий процесс, чтобы не повторяться с описанием разных возможностей в каждой новой задаче.
* **[MCP (Model Context Protocol)](https://modelcontextprotocol.io/)** — Протокол для расширения контекста и интеграции внешних инструментов и сервисов с ИИ-агентами. Позволяет подключать дополнительные источники данных и функциональность.

Эти инструменты полезны для разработки собственного harness (набора инструментов и конфигураций) под большинство задач и рабочих процессов в различных ИИ-агентах, включая Roo Code.
